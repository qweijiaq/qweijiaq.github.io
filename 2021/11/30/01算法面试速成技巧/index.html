01算法面试速成技巧

算法 面试
九章算法
Publish Date:   2021-11-30

面试官眼中的求职者

HR主要通过你的衣着、谈吐和外貌来看你；技术面试官则是通过你的代码来看你。

写代码不超过3个月

为什么面试官会认为上述代码是由代码基础不足三个月的人写出的？

  1. 变量含义不明。好的代码要求每一个（至少重要的）变量和函数含义明了，最好直接能通过命名来得知它的功能。
  2. 嵌套过多。嵌套越多也就意味着代码出错的可能性越大，可读性也越差，一般代码的嵌套最好不要超过三层，如果实在需要更多层的嵌套，可以将其封装在一个函数里。

代码从没被人review过

如何在公司获得提升？

很多人认为在公司里面多做项目就可以获得巨大的提升，这只是一个半对的答案。事实上，当我们的代码被前辈（更高级别的程序员）review，并给你feedback一些建议，我们根据这些反馈做出积极的改变时，往往能够得到最大的提升。与学习编程时不同，工程中的编程更加考验我们的工程思维，一段好懂的代码往往比一段简短的代码重要的多。当我们写的代码从来没有被review时，我们往往只在意我们所写的代码是否正确，而去忽视了它是否好理解，是否能够更加的优化。这时，前辈的某些建议可能是醍醐灌顶的，他可能帮你解决了你一辈子不能解决甚至不会发现的问题，这种时候带来的提升很可能是质变的。

就拿上图来说，grid[i][j] = 1 or 2的写法没有语法错误，运行起来也正常，但是如果加上：

class GridType:
    PEOPLE = 0
    ZOMBIE = 1
    WALL = 2

然后将代码改为grid[i][j] = GridType.ZOMBIE or GridType.WALL，虽然增加了几行代码，但会极大增加可读性，让面试官觉得你是一个具有工程思维的人。

写代码经验太少

写代码经验太少也是一个需要解决的问题，这需要平时不断的实战、不断的思考、不断的debug。如上图就是一段有问题的代码。它使用到了or这一逻辑运算符，我们知道and和or都是遵循短路原则的，对于这里面的or来说，如果前面的i < len(list)成立，就直接默认这个条件判断是True的，不会去判断后面的条件j < len(list2)是否成立，从而可能导致程序出现bug。

缺乏项目经验
  • 代码耦合度太高，逻辑交杂在一起
    • 难维护
    • 难读懂
    • 易出错

解决办法：解耦合(Decouple)

将上图的算法分解为三个步骤：

  1. 找到所有点
  2. 复制所有点
  3. 复制所有边
def cloneGraph(self, node):
    if not node:
        return None
    
    # step 1: find nodes
    nodes =. self.find_nodes_by_bfs(node)
    # step 2: copy nodes
    mapping = self.copy_nodes(nodes)
    # step 3: copy edges
    self.copy_edges(nodes, mapping)
    
    return mapping[node]

def find_nodes_by_bfs(self, node):
    queue = collections.deque([node])
    visited = set([node])
    while queue:
        curt_node = queue.popleft()
        for neighbor in curt_node.neighbors:
            if neighbor in visited:
                continue
            visited.add(neighbor)
            queue.append(neighbor)
    return list(visited)

def copy_nodes(self, nodes):
    mapping = {}
    for node in nodes:
        mapping[node] = UndirectedGraphNode(node.label)
    return mapping

def copy_edges(self, nodes, mapping):
    for node in nodes:
        new_node = mapping[node]
        for neighbor in node.neighbors:
            new_neighbor = mapping[neighbor]
            new_node.neighbors.append(new_neighbor)

这样修改的代码在结构上比较清晰明了，阅读起来也没有压力，最重要的是降低了代码之间的耦合性。

重复代码和全局变量的问题会在后面的课程中学习到。

代码质量(Code Quality)

你认为的代码质量：

  • 代码要加注释（这一点本身没错）

注释固然重要，但是使用含义清晰的变量名命名 + 简单易读的处理逻辑更为重要。我们写注释的目的就是让人看懂我们所写代码的含义，有时候一个好的命名规范可以让他人仅凭变量名或者函数名就能知道它要实现的功能。

  • 代码越短越好

事实上，通过适当的子函数化的代码包装，并且多加空行，虽然在观感上代码更长了，但是能够让你的代码更加易读、易维护，且不易错。

好的代码质量真的那么重要么？

回答是肯定的，代码就像一件艺术品，越是高级的程序员，越有代码洁癖越是高级别的面试官，越在意你的代码质量。

  • 拥有好的代码质量，会让面试官在心里为你默默加分；如果质量很差，面试官会在心里为你默默扣分。最终是否导致Hire / No Hire，就是一个量变引起质变的问题。
  • 拥有好的代码质量，还能够让你的代码少出BUG。你以为只需要细心就可以不出BUG？但是通过子函数化、避免全局变量等手段可以让你出 BUG 的概率大大降低。

面试评价体系

  • 逻辑思维能力(Logicality)
    • 是否能很快的想到一个Working Solution
    • 是否能够在面试官点出问题后优化自己的Solution
  • 代码质量(Code Quality)
    • 代码到底写完没有
    • 代码风格好不好
      • 可读性
      • 变量名、函数名命名
      • 空格和空行的正确使用
  • 特殊情况/边界情况(Edge Cases)
  • Bug Free

沟通能力

把面试官当作Co-worker，而不是考官，让面试官愿意和你一起工作。

几个面试沟通法则

做一个题之前，先沟通清楚，得到面试官肯定，再开始写代码，写完以后再解释

  • 不要闷头写

  • 不要一边写一边解释太多（容易写不完）

可以要提示，经过提示做出来的题，也是可以拿到Hire的

  • 但是先自己努力想一下，别太容易放弃，容易让人觉得不会主动思考问题

不要和面试官吵架

  • 面试官大概率是对的，面试官是带着答案来面试你的

  • 不同意见在大部分情况下，很可能是你自己想错了

会就会，不会就不会，不要遮遮掩掩，坦诚自信很重要

  • 容易让人觉得和你沟通“不顺畅”

  • 做过的题就说做过，不要故意说没做过

  • 因为他既然已经怀疑你做过了，即使你说没有，他也无法打消这个顾虑，还不如让他换

算法面试九步法

理解问题、复述问题、澄清问题、初步想法、实例分析、书写代码、测试检验、评判性能、优化解法

刷题

刷题刷到什么程度去面试才够？

你永远没有觉得自己准备好的那一天！LintCode 可以帮你解决烦恼！

刷题的三个维度：

  • 算法能力
    • LintCode CAT(Code Ability Test)可以帮你评估算法能力
    • 想拿到Offer，至少要达到黄金
  • Bug Free能力
    • 每道题的平均提交次数可以评估你的Bug Free能力，一般情况下平均每道题不超过3次
  • 题量
    • 这是很多人只关心的维度，但不能否认它是最弱的维度

面试算法 VS. 算法

如果你还在看《算法导论》？赶紧扔掉 ——我宁可你看的是《Cracking The Coding Interview》

也请不要去看普林斯顿的算法公开课 ——很多内容面试依然不考，或考得很少

为什么？ —— 面试算法不等于算法

当然，这是一种应试思维，如果想在算法上有些学术的研究，《算法导论》还是少不了的，我自己出于兴趣也会时不时拿出来翻几页。

也就是说面试的算法是算法体系里面最“虚”的部分。算法的种类繁多，但真正在面试中会考到的往往十分有限，所以必须给算法面试“划考点”。

上述图片中，越红考得越多，灰色不考或者出现概率低于千分之一。

线段数值得关注一下，因为它具有通用性，知道了线段树，平衡排序二叉树、跳跃表等不成问题。

算法面试知识点及考频：

去 FMAG 是否一定要刷难题？

  • Google（活少钱多）

    • 要刷难题较高的题目
    • 算法考察范围很广，特别喜欢DP和红黑树 Red-black Tree，线段树 Segment Tree

  • Facebook（活多钱多）, Amazon（活多钱少）, Microsoft（活少钱少） 等大厂

    • 刷中等题就够了，更加要求bug free

  • 其他中小厂

    • 刷中等题就够了
    • 算法面试考察范围很窄
    • Binary Tree, LinkedList, String, Array这些基本数据结构相关的题掌握即可

不同的岗位算法面试难度不同吗？

是的，不同，越后端越难，越前端越简单。

简单 Web Frontend Engineer / Mobile Engineer / Data Engineer
中等 Software Development Engineer / Site Reliability Engineer (运维)
难 Infrastructure Engineer
超难 Machine Learning Engineer / Data Scientist

其他面试

技术岗除了面算法题，还有其他哪些类型的面试？

  • 系统设计 System Design / Architechture Design

    • 系统设计System Design是一种非常常见的面试形式，通常出现在后端相关的岗位中，前端工程师和数据工程师一般不太考察，应届生也可能会考一些简单的系统设计。

  • 面向对象设计 Objected Oriented Design

    • 互联网公司一般考System Design多，软件公司一般考OOD；互联网公司 = Facebook / Google / Amazon之流；软件公司 = Microsoft / Apple / Oracle之流；应届生New Grad通常会考OOD多于System Design。

  • 行为面试 Behavior Question

    • BQ面试是现在各大公司技术岗位几乎必须面的一轮面试，通常是其他非技术岗位的人面试你，比如 HR, Product/Project Manager, Designer。

  • 简历面试 Experience Interview
